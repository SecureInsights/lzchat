# 音频/视频通话修复闭环报告 - lzchat

**日期**: 2026-07-08  
**范围**: WebCodecs + WebSocket E2EE relay 通话实现  
**状态**: 审计项已按优先级修复，仍建议后续补浏览器真实媒体 E2E。

## 当前通话架构

```text
getUserMedia
  -> WebCodecs encode
  -> call-media payload
  -> 独立 media ratchet sealPayload(E2EE)
  -> WebSocket / Durable Object blind relay
  -> 独立 media ratchet openPayload
  -> WebCodecs decode
```

服务端仍只看到：

- `roomId`
- `from/to`
- envelope kind
- 密文大小
- 连接与频率元数据

服务端仍看不到：

- 音频明文
- 视频明文
- 媒体 payload 明文
- 通话内容

## 修复清单

### H1: 通话媒体与聊天共用 ratchet

**状态**: 已修复

修复内容：

- `PairSession` 新增 `mediaSendCK` / `mediaRecvCK`。
- 每个 peer 新增 `mediaSendRatchet` / `mediaRecvRatchet`。
- `call-media` 改用独立 media ratchet。
- 普通 text/file/profile/private/call-signal 继续使用原消息 ratchet。

影响：

- 媒体高频 seq 不再消耗聊天消息 seq。
- 媒体乱序/跳包不会挤占普通聊天 skipped key window。

涉及文件：

- `client/src/crypto/handshake.ts`
- `client/src/app.ts`

### H2: `call-media` 可用于高带宽/高 CPU DoS

**状态**: 已修复

修复内容：

- 服务端和 Worker 现在先 `validateRelayEnvelope()`，再根据已验证的 `relay.kind` 决定限速桶。
- 新增 media byte budget：`6MB / 10s`。
- 客户端新增 per-peer media byte budget。
- 客户端在没有 active call 或对端不匹配时，解密前丢弃 `call-media`。
- 视频 chunk 上限降为 `128KB`。
- 音频 chunk 上限降为 `16KB`。
- validator 按 audio/video 分别限制 `bytes` 长度。

涉及文件：

- `server/src/relay.ts`
- `server/src/validators.ts`
- `worker/src/index.ts`
- `worker/src/validators.ts`
- `client/src/app.ts`
- `client/src/protocol/validator.ts`

### H3: decoder codec 过宽且缺少错误边界

**状态**: 已修复

修复内容：

- `call-media` codec 改为白名单：
  - video: `vp8`, `vp09.00.10.08`, `avc1.42E01E`
  - audio: `opus`
- `VideoDecoder.configure/decode` 加 try/catch。
- `AudioDecoder.configure/decode` 加 try/catch。
- decoder output callback 加错误边界。

涉及文件：

- `client/src/protocol/validator.ts`
- `client/src/app.ts`

### H4: caller 收到 answer 后启动 publisher 没有失败恢复

**状态**: 已修复

修复内容：

- `call-answer` 分支对 `startEncodedPublisher()` 加 try/catch。
- 启动失败时：
  - 清理本地 tracks/encoder/decoder。
  - 清空 call state。
  - 显示系统提示。
  - 向对端发送 `call-end`，reason=`media-failed`。

涉及文件：

- `client/src/app.ts`

### M2: 来电无超时

**状态**: 已修复

修复内容：

- 新增 `CALL_INCOMING_TIMEOUT_MS = 60_000`。
- 来电 60 秒未接自动结束。

- 接听、拒绝、挂断、离开房间时清理 timeout。

涉及文件：

- `client/src/app.ts`

### M3: 视频首帧遮罩可能不会消失

**状态**: 已修复

修复内容：

- 首个 video keyframe 成功进入 decoder 后触发一次 `renderChat()`。
- 只在第一次 keyframe 触发，避免每个 keyframe 重绘导致闪烁。

涉及文件：

- `client/src/app.ts`

### M4: 音频播放队列没有延迟上限

**状态**: 已修复

修复内容：

- 新增 `CALL_MAX_AUDIO_QUEUE_DELAY_SEC = 0.8`。
- 如果 `nextAudioTime` 超过当前时间太多，重置播放起点，避免延迟无限累积。

涉及文件：

- `client/src/app.ts`

### M5: 媒体发送失败被静默吞掉

**状态**: 已修复

修复内容：

- 新增连续媒体发送失败计数。
- 成功发送后重置失败计数。
- 连续失败达到阈值后：
  - 显示系统提示。
  - 自动结束通话。
  - 通知对端。

涉及文件：

- `client/src/app.ts`

这些不是本轮发现的代码漏洞，但属于通话功能成熟度问题：

- 增加 Playwright fake media E2E，覆盖两端语音/视频按钮、权限流程和挂断流程。
- 增加真实浏览器手动测试矩阵：Chrome/Edge/Firefox，`localhost` 与 HTTPS Worker 环境。
- 后续如果要做多人通话，应独立设计 SFU/MoQ/mesh 策略，不要直接复用当前 1:1 UI。
